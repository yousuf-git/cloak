# Cloak — Core Logics

> How a secret actually travels through Cloak: from a keystroke in the desktop app, through
> client-side encryption, across the wire, into MongoDB — and back out again on reveal. Every step
> below is traced to the real code with clickable `file:line` links, and each transformation is shown
> with a concrete worked example.

This is a ground-truth companion to [`PLAN.md`](../PLAN.md). Where the running code diverges from the
plan, this document describes **the code**, and flags the divergence in
[§10 Divergences from the plan](#10-divergences-from-the-plan).

---

## 1. The one idea that explains everything: envelope encryption

Cloak is **zero-knowledge**: the server never sees the master password or any secret plaintext. This
is achieved with a two-layer key hierarchy — a per-user data key that is itself encrypted.

```
 master password ──Argon2id──► MasterKey ─────────┐  (never leaves device)
        │                                          │
        └───────Argon2id──────► authHash ──► server: argon2id(authHash) = password_hash
                                                    │
        VaultDEK (32 random bytes) ◄── unwrap ──────┤
              │                     wrap             │
              │        wrappedDEK = XChaCha20(MasterKey, VaultDEK)  ──► stored in DB
              │
              └── encrypts every secret field: XChaCha20(VaultDEK, plaintext)
```

- **MasterKey** — derived from the password, wraps the DEK. Lives only in Rust memory. Never stored,
  never sent.
- **VaultDEK ("DEK")** — a random 256-bit key that *actually* encrypts every field. Stored only in
  its wrapped form (`wrappedDEK`). Unwrapped into Rust memory at login.
- **authHash** — a *different* derivation of the same password, sent to the server purely to prove
  identity. The server hashes it *again* before storage, so a DB leak yields no replayable credential.

Why a DEK layer at all? Rotating the master password (or recovery) only re-wraps 32 bytes — it never
re-encrypts the whole vault. See [`PLAN.md` §3.1](../PLAN.md).

**Where the crypto runs:** all MasterKey/DEK operations execute in the **Rust/Tauri core**
([`desktop/src-tauri/src/commands/crypto.rs`](../desktop/src-tauri/src/commands/crypto.rs)). The
React webview only ever holds ciphertext strings and, transiently, freshly-decrypted plaintext for
display. The raw DEK bytes never cross the webview boundary.

### 1.1 Terms & where each key lives

| Thing | Bytes | Derived / generated by | Lives where | Sent to server? |
|-------|-------|------------------------|-------------|-----------------|
| `crypto_salt` | 16 rand → base64 | [`kdf.rs:58`](../desktop/src-tauri/src/crypto/kdf.rs#L58) | DB (`users.crypto_salt`), public | Yes (own field) |
| MasterKey | 32 | Argon2id + `cloak:mk` XOR | Rust session memory | **Never** |
| authHash | 32 → base64 | Argon2id + `cloak:auth` XOR | transient | Yes (then re-hashed) |
| `password_hash` | argon2id string | server [`hashing.ts:9`](../api/src/lib/hashing.ts#L9) | DB (`users.password_hash`) | — (server-only) |
| VaultDEK | 32 rand | [`dek.rs:8`](../desktop/src-tauri/src/crypto/dek.rs#L8) | Rust session memory | **Never** (only wrapped) |
| `wrappedDEK` | base64 | `XChaCha20(MasterKey, DEK)` | DB (`users.wrappedDEK`) | Yes (opaque) |
| recovery key | 160-bit Crockford | [`kdf.rs:93`](../desktop/src-tauri/src/crypto/kdf.rs#L93) | shown once, then wiped | **Never** |
| `recovery_wrappedDEK` | base64 | `XChaCha20(recoveryWK, DEK)` | DB (`users.recovery_wrappedDEK`) | Yes (opaque) |

## 2. The two ciphers

Everything below is built from two primitives. Learn these and every flow becomes obvious.

### 2.1 Field cipher — XChaCha20-Poly1305 (AEAD)

Used for: DEK wrapping, every credential/API-key/backup-code field, and the wrapped dotenvx key.

- **Seal** ([`aead.rs:14`](../desktop/src-tauri/src/crypto/aead.rs#L14)): generate a random 24-byte
  nonce, encrypt+authenticate, output `base64(nonce ‖ ciphertext ‖ tag)`.
- **Open** ([`aead.rs:31`](../desktop/src-tauri/src/crypto/aead.rs#L31)): split off the 24-byte
  nonce, decrypt, verify the 16-byte Poly1305 tag (tampering ⇒ error).

```
seal(key, "hunter2")
  nonce  = 24 random bytes
  ct+tag = XChaCha20Poly1305(key, nonce, "hunter2")     # 7 bytes ct + 16 bytes tag
  output = base64( nonce ‖ ct ‖ tag )                    # 24+7+16 = 47 bytes → 64 base64 chars
         = "n0pQ…KcQ=="   (opaque; a new random nonce every call ⇒ different ciphertext each save)
```

The wire/stored layout for **any** field is always this `base64(nonce‖ct‖tag)` string. The server
treats it as an opaque `String`.

### 2.2 `.env` cipher — dotenvx-compatible ECIES over secp256k1

Used for: `.env` file values, so the output is decryptable by the real `dotenvx` CLI.
([`dotenvx_compat.rs`](../desktop/src-tauri/src/crypto/dotenvx_compat.rs), wrapping the `dotenvx`
crate.)

- Each env file gets its own **secp256k1 keypair** ([`dotenvx_compat.rs:13`](../desktop/src-tauri/src/crypto/dotenvx_compat.rs#L13)).
- The **public key** encrypts each value → `encrypted:BASE64...` ([`:21`](../desktop/src-tauri/src/crypto/dotenvx_compat.rs#L21)).
- The **private key** is needed to decrypt — and it is itself sealed with the field cipher (§2.1)
  under the DEK, so it can live in the DB safely.

```
KEY=value   ──encrypt_env_value(pubkey)──►   KEY="encrypted:BAllx3…Q=="
```

## 3. Flow A — Signup (input → keys → DB)

**Trigger:** user submits email + master password on the signup form; store action
[`auth.ts:101`](../desktop/src/stores/auth.ts#L101).

### Step-by-step

1. **Client derives all crypto material in Rust** — one call,
   [`crypto.prepareSignup`](../desktop/src/lib/tauri-crypto.ts#L39) →
   [`crypto_prepare_signup`](../desktop/src-tauri/src/commands/crypto.rs#L48):
   - generate `crypto_salt` (16 bytes) — [`kdf.rs:58`](../desktop/src-tauri/src/crypto/kdf.rs#L58)
   - `MasterKey = derive(pw, salt, "cloak:mk")` — [`kdf.rs:45`](../desktop/src-tauri/src/crypto/kdf.rs#L45)
   - `authHash  = base64(derive(pw, salt, "cloak:auth"))` — [`kdf.rs:53`](../desktop/src-tauri/src/crypto/kdf.rs#L53)
   - `DEK = 32 random bytes`; `wrappedDEK = seal(MasterKey, DEK)` — [`crypto.rs:56`](../desktop/src-tauri/src/commands/crypto.rs#L56)
   - `recovery_key` (160-bit Crockford); `recovery_wrappedDEK = seal(recoveryWK, DEK)` — [`crypto.rs:61`](../desktop/src-tauri/src/commands/crypto.rs#L61)

2. **Client POSTs only hashes + ciphertext** — [`api.signup`](../desktop/src/lib/api.ts#L128) →
   `POST /api/v1/auth/signup`.

3. **Server re-hashes and stores** — [`auth.service.ts:23`](../api/src/services/auth.service.ts#L23):
   `password_hash = argon2id(authHash)` ([`hashing.ts:9`](../api/src/lib/hashing.ts#L9)), then
   `User.create({...})` ([`user.model.ts`](../api/src/models/user.model.ts)). Sends an email-verify OTP.

4. **Client shows the recovery key once**, then wipes it from memory
   ([`auth.ts:124`](../desktop/src/stores/auth.ts#L124)).

### The KDF, precisely

[`kdf.rs:23`](../desktop/src-tauri/src/crypto/kdf.rs#L23) — Argon2id, `m=19456 KiB, t=2, p=1`,
32-byte output, then **domain separation** by XOR-ing the associated-data label (`cloak:mk` /
`cloak:auth` / `cloak:rk`) cyclically over the output. Same password + salt ⇒ two *different* keys
for the MasterKey vs the authHash context.

### Worked example

```
email       = "dev@example.com"
password    = "correct-horse-battery-staple"

crypto_salt        = "sQ8f3Kd9Lm2pR7xA=="                     # base64 of 16 random bytes
MasterKey (hex)    = 4f1c… (32 bytes, stays in Rust)
authHash           = "Yb3K…Qe0="                              # base64 of 32 bytes
DEK (hex)          = a90b… (32 random bytes, stays in Rust)
wrappedDEK         = "Zt7…="                                  # base64(nonce‖ct‖tag), 72 bytes
recovery_key       = "9F3K-2M7Q-8XZ4-…-VW10"                  # 8×4 Crockford groups, shown once
recovery_wrappedDEK= "Pa2…="

POST /auth/signup
{ email, authHash:"Yb3K…Qe0=", cryptoSalt:"sQ8f3Kd9Lm2pR7xA==",
  wrappedDEK:"Zt7…=", recoveryWrappedDEK:"Pa2…=" }

users doc written:
{ email:"dev@example.com",
  password_hash:"$argon2id$v=19$m=…$…",   # argon2id(authHash) — NOT the password, NOT the authHash
  crypto_salt:"sQ8f3Kd9Lm2pR7xA==",
  wrappedDEK:"Zt7…=", recovery_wrappedDEK:"Pa2…=",
  is_verified:false }
```

The password and both raw keys (MasterKey, DEK) **never appear** in the request or the DB.

## 4. Flow B — Login + 2FA + session unlock

**Model:** *client derives, server verifies.* The KDF runs on-device; only the server decides
pass/fail. Store action [`auth.ts:142`](../desktop/src/stores/auth.ts#L142).

1. **Prelogin (salt only)** — [`api.prelogin`](../desktop/src/lib/api.ts#L122) → `POST /auth/prelogin`
   → [`auth.service.ts:52`](../api/src/services/auth.service.ts#L52) returns
   `{ crypto_salt, two_factor_enabled }`. Unknown emails get a **deterministic fake salt**
   ([`:60`](../api/src/services/auth.service.ts#L60)) so account existence isn't leaked.

2. **Re-derive authHash locally** — [`crypto.deriveAuthHash`](../desktop/src/lib/tauri-crypto.ts#L44)
   → [`crypto_derive_auth_hash`](../desktop/src-tauri/src/commands/crypto.rs#L115). Password never
   leaves the device.

3. **Server verifies** — `POST /auth/login` → [`auth.service.ts:71`](../api/src/services/auth.service.ts#L71):
   `argon2.verify(password_hash, authHash)` ([`hashing.ts:13`](../api/src/lib/hashing.ts#L13)). On
   mismatch → generic `Invalid email or password`.

4. **If 2FA enabled** → server emails an OTP, returns `{ status: '2fa_required' }`. Client submits
   `POST /auth/2fa` → [`verifyTwoFactor`](../api/src/services/auth.service.ts#L97).

5. **On success** the server returns `{ accessToken, refreshToken, wrappedDEK }`.

6. **Client unlocks the session** — [`crypto.unlockSession`](../desktop/src/lib/tauri-crypto.ts#L61)
   → [`crypto_unlock_session`](../desktop/src-tauri/src/commands/crypto.rs#L127):
   re-derive MasterKey, `DEK = unwrap_dek(MasterKey, wrappedDEK)`, store **both** in the Rust session
   holder ([`:137`](../desktop/src-tauri/src/commands/crypto.rs#L137)). The vault is now usable.

```
server → { accessToken:"eyJ…", refreshToken:"rt_…", wrappedDEK:"Zt7…=" }
unlockSession(pw, salt, "Zt7…="):
   MasterKey = derive(pw, salt, "cloak:mk")
   DEK       = open(MasterKey, "Zt7…=")        # 32 bytes recovered into Rust memory
   session.set_master_key(MasterKey); session.set_dek(DEK)
```

**Tokens** live in webview memory only ([`api.ts:26`](../desktop/src/lib/api.ts#L26)); a 401 triggers
one transparent refresh ([`api.ts:109`](../desktop/src/lib/api.ts#L109)). **Remember-Me** persists the
DEK + refresh token in the OS keychain via Rust ([`crypto.rs:308`](../desktop/src-tauri/src/commands/crypto.rs#L308))
— the DEK still never crosses the webview.

## 5. Flow C — Create a credential (input → DB) ⭐ the headline example

This is the "when a user inputs, what happens until it's stored" walkthrough.

**UI:** the New Credential modal ([`CredentialsPage.tsx:217`](../desktop/src/pages/CredentialsPage.tsx#L217)).
User types `name`, `url`, `username`, `password`, `note` and clicks Create.

### The pipeline

```
 [modal submit]                 CredentialsPage.tsx:160
      │  values = { name, url, username, password:"hunter2", note }
      ▼
 encrypt(password)              useVaultCrypto.ts:14  ──► crypto_encrypt_field  (Rust)
      │  password → seal(DEK, "hunter2") → "n0pQ…KcQ=="
      ▼
 payload = { name, url, note,
             username:"octocat",        # PLAINTEXT — username is searchable metadata, not a secret
             password:"n0pQ…KcQ==" }     # ciphertext
      ▼
 create(payload)               hooks/vault.ts:33  ──► vaultApi.createCred
      │
      ▼
 POST /api/v1/vault/creds      api.ts:254   (Authorization: Bearer <access token>)
      ▼
 requireAuth                   require-auth.ts:15   verifies JWT → req.user.sub = userId
      ▼
 validate(credCreateSchema)    vault.routes.ts:28   Zod body check
      ▼
 createCred controller         vault.controller.ts:20   uid(req) = req.user.sub
      ▼
 vault.service.createCred      vault.service.ts:15   Cred.create({ ...body, user_id })
      ▼
 MongoDB `creds` collection    cred.model.ts:17
```

### Key facts at each hop

- **Encryption happens in the webview→Rust boundary, before the network.** The only crypto the
  frontend does is *invoke Rust*; see [`useVaultCrypto.ts:14`](../desktop/src/hooks/useVaultCrypto.ts#L14)
  (real mode) which calls [`crypto.encryptField`](../desktop/src/lib/tauri-crypto.ts#L67).
- **The password field is the only encrypted field.** `name`, `url`, `note` and crucially `username`
  are stored **plaintext** — the model comments say so explicitly
  ([`cred.model.ts:8`](../api/src/models/cred.model.ts#L8)) — because instant client-side search
  requires unencrypted metadata (see [`PLAN.md` §6](../PLAN.md)).
- **Tenant scoping is server-enforced.** The client never sends `user_id`; the controller derives it
  from the verified JWT ([`vault.controller.ts:8`](../api/src/controllers/vault.controller.ts#L8)) and
  the service stamps it onto the document ([`vault.service.ts:16`](../api/src/services/vault.service.ts#L16)).
- **The server stores ciphertext blindly.** `password` is a `required: true` `String`
  ([`cred.model.ts:24`](../api/src/models/cred.model.ts#L24)) — the API has no key and cannot read it.
- **An audit-log row is written** — action `cred:create`, **metadata only**, never the value
  ([`vault.controller.ts:22`](../api/src/controllers/vault.controller.ts#L22)).

### What lands in MongoDB

```json
// creds collection
{
  "_id": "665f…",
  "user_id": "6650…",          // from JWT, not from the client body
  "name": "GitHub",
  "url": "https://github.com",
  "username": "octocat",        // plaintext
  "password": "n0pQ…KcQ==",     // base64(nonce‖ct‖tag) — opaque to the server
  "note": "personal",
  "created_at": "2026-07-13T…",
  "updated_at": "2026-07-13T…"
}
```

> **API keys, backup codes** follow the identical pattern: `key` / `encrypted_code` are encrypted with
> the same `crypto_encrypt_field`; `label` and platform `name` stay plaintext. See
> [`vault.service.ts:35`](../api/src/services/vault.service.ts#L35) (API keys) and
> [`:70`](../api/src/services/vault.service.ts#L70) (backup codes). Projects are **embedded on the
> user document** and fully plaintext ([`vault.service.ts:110`](../api/src/services/vault.service.ts#L110)).

## 6. Flow D — Retrieve & reveal a credential (DB → display)

### 6.1 List (fetch ciphertext)

On mount, [`useCreds`](../desktop/src/hooks/vault.ts#L7) runs a TanStack Query →
`GET /api/v1/vault/creds` → [`listCreds`](../api/src/controllers/vault.controller.ts#L19) →
[`Cred.find({ user_id }).sort({ created_at: -1 }).lean()`](../api/src/services/vault.service.ts#L11).

The query is **scoped to `user_id`** — a user can only ever read their own rows. The response
contains the ciphertext `password` exactly as stored. Nothing is decrypted server-side.

### 6.2 Render masked

Each row renders the password through [`SecretField`](../desktop/src/components/ui/SecretField.tsx#L134),
wired at [`CredentialsPage.tsx:140`](../desktop/src/pages/CredentialsPage.tsx#L140):

```tsx
<SecretField cipher={cred.password} reveal={() => decrypt(cred.password)} maskLength={20} />
```

By default it shows a **truncated preview of the ciphertext itself** (`maskedPreview`,
[`SecretField.tsx:128`](../desktop/src/components/ui/SecretField.tsx#L128)) — the user sees encrypted
characters, reinforcing that nothing is decrypted until asked.

### 6.3 Reveal on demand (lazy decrypt)

Clicking the eye calls `reveal()` → `decrypt(cred.password)`
([`useVaultCrypto.ts:20`](../desktop/src/hooks/useVaultCrypto.ts#L20)) →
[`crypto_decrypt_field`](../desktop/src-tauri/src/commands/crypto.rs#L155) →
`open(DEK, ciphertext)` → plaintext. The plaintext is cached in local component state **only while
revealed**, and a scramble animation transitions ciphertext↔plaintext
([`SecretField.tsx:161`](../desktop/src/components/ui/SecretField.tsx#L161)).

```
"n0pQ…KcQ=="  ──crypto_decrypt_field──►  open(DEK, …)  ──►  "hunter2"
                                          (Rust: nonce split, Poly1305 verified, UTF-8 decoded)
```

Copy does the same lazy `resolve()` then writes to the clipboard
([`SecretField.tsx:174`](../desktop/src/components/ui/SecretField.tsx#L174)). Hiding re-masks and
drops the cached plaintext.

**Editing** re-decrypts to prefill the form ([`CredentialsPage.tsx:239`](../desktop/src/pages/CredentialsPage.tsx#L239)),
then re-encrypts on save — a fresh nonce means the stored ciphertext changes even for an unchanged
password.

## 7. Flow E — `.env` files (dotenvx)

Env files are the exception to per-field crypto: the **whole file** is stored as one dotenvx blob,
and "reveal" decrypts the entire document at once (there are no field-wise encrypted values in the
DB — see [`PLAN.md` §1](../PLAN.md)).

### 7.1 Import a plaintext `.env` (create)

Orchestrated by [`useEnvFiles.importPlain`](../desktop/src/hooks/useEnvFiles.ts#L63).

1. `crypto.envEncryptNew(plaintext)` → [`crypto_env_encrypt_new`](../desktop/src-tauri/src/commands/crypto.rs#L206):
   - generate a fresh secp256k1 keypair,
   - `encrypt_env_file` — rewrite every `KEY=value` to `KEY="encrypted:…"`, prepend a
     `DOTENV_PUBLIC_KEY` header, preserve comments/blanks verbatim
     ([`dotenvx_compat.rs:34`](../desktop/src-tauri/src/crypto/dotenvx_compat.rs#L34)),
   - **wrap the private key with the DEK**: `wrapped_key_b64 = seal(DEK, privKey)`
     ([`crypto.rs:213`](../desktop/src-tauri/src/commands/crypto.rs#L213)) — so Remember-Me restore
     can still decrypt later.
2. `vaultApi.createEnvFile({ …, encrypted_dotenvx_key: wrapped_key_b64, content_b64, variable_count })`
   — the blob is base64-transported ([`useEnvFiles.ts:64`](../desktop/src/hooks/useEnvFiles.ts#L63)).
3. Server decodes and stores — [`env-file.service.ts:28`](../api/src/services/env-file.service.ts#L28);
   model [`env-file.model.ts`](../api/src/models/env-file.model.ts).

```
plaintext .env:                    stored content (dotenvx wire format):
  # db                               #/ dotenvx encrypted (Cloak) /
  DATABASE_URL=postgres://x          DOTENV_PUBLIC_KEY="03a1…"          # secp256k1 pubkey, hex
  API_KEY=sk-123                     # db
                                     DATABASE_URL="encrypted:BAll…=="
                                     API_KEY="encrypted:BM2p…=="

DB env-file doc:
  { label, tag:"Local", project_id,
    content:"<the block above>",                # opaque secret blob
    encrypted_dotenvx_key:"seal(DEK, privKey)", # null ⇒ view-only import
    variable_count:2 }
```

If the user imports an **already-encrypted** file, the private key is optional
([`importEncrypted`](../desktop/src/hooks/useEnvFiles.ts#L76)): supply it → it's wrapped and stored;
leave blank → `encrypted_dotenvx_key: null` and the file is **view-only** (can be shown/copied in
encrypted form but never decrypted).

### 7.2 View → decrypt → edit → save

The [`EnvViewer`](../desktop/src/components/env/EnvViewer.tsx) drives this:

- **Load raw:** `GET /env-files/:id/raw` → [`getEnvRaw`](../api/src/services/env-file.service.ts#L44)
  streams back the encrypted blob (audit action `env:view`). Shown as-is.
- **Decrypt** ([`EnvViewer.tsx:50`](../desktop/src/components/env/EnvViewer.tsx#L50)):
  [`useEnvFiles.decrypt`](../desktop/src/hooks/useEnvFiles.ts#L98) → [`crypto_env_decrypt`](../desktop/src-tauri/src/commands/crypto.rs#L228)
  — unwrap the dotenvx private key with the DEK, then `decrypt_env_file` reverses every `encrypted:…`
  back to plaintext ([`dotenvx_compat.rs:64`](../desktop/src-tauri/src/crypto/dotenvx_compat.rs#L64)).
  Disabled when no key is stored ([`EnvViewer.tsx:20`](../desktop/src/components/env/EnvViewer.tsx#L20)).
- **Edit + save** ([`EnvViewer.tsx:78`](../desktop/src/components/env/EnvViewer.tsx#L78)):
  [`saveEdit`](../desktop/src/hooks/useEnvFiles.ts#L107) → [`crypto_env_encrypt_existing`](../desktop/src-tauri/src/commands/crypto.rs#L247)
  re-encrypts the edited plaintext under the **same public key** (extracted from the header), then
  `PATCH /env-files/:id` overwrites `content` + `variable_count`
  ([`env-file.service.ts:58`](../api/src/services/env-file.service.ts#L58)).

Because encryption only needs the public key, editing requires no unwrapping beyond the initial
decrypt — and the file stays decryptable by anyone holding the (still-wrapped) private key.

## 8. Flow F — Zero-knowledge account recovery

The recovery envelope (`recovery_wrappedDEK`, written at signup) lets a user recover the vault
**without** the master password. Store actions [`auth.ts:209`–273](../desktop/src/stores/auth.ts#L209).

1. **Start** — `POST /auth/recovery/start` emails an OTP; always "succeeds" to avoid enumeration
   ([`auth.service.ts:151`](../api/src/services/auth.service.ts#L151)).
2. **Verify** — `POST /auth/recovery/verify` returns `{ crypto_salt, recovery_wrappedDEK,
   recoveryToken }` ([`auth.service.ts:165`](../api/src/services/auth.service.ts#L165)).
3. **Reset (all crypto client-side)** — [`crypto_recovery_reset`](../desktop/src-tauri/src/commands/crypto.rs#L79):
   - `recoveryWK = derive(recoveryKey, salt, "cloak:rk")`; `DEK = unwrap(recoveryWK, recovery_wrappedDEK)`
     — wrong key ⇒ clean error ([`crypto.rs:88`](../desktop/src-tauri/src/commands/crypto.rs#L88)),
   - generate a **new salt**, derive a new MasterKey + authHash from the new password,
     re-wrap the DEK under both the new MasterKey and a new recovery envelope,
   - establish the session immediately ([`crypto.rs:101`](../desktop/src-tauri/src/commands/crypto.rs#L101)).
4. Server rotates `password_hash`, `crypto_salt`, both envelopes, and **revokes all sessions**
   ([`auth.service.ts:196`](../api/src/services/auth.service.ts#L196)).

The DEK is preserved across the reset, so **existing vault ciphertext stays valid** — only its
wrapping keys change.

## 9. Data-at-rest summary

What an attacker with full DB read access actually sees:

| Collection | Encrypted (opaque) | Plaintext (searchable metadata) |
|-----------|--------------------|--------------------------------|
| `users` | `wrappedDEK`, `recovery_wrappedDEK`, `password_hash`¹ | `email`, `crypto_salt`, `two_factor_enabled`, `projects[]` |
| `creds` | `password` | `name`, `url`, `username`, `note` |
| `api-keys` | `key` | `label`, `url`, `note` |
| `platform` | `backup_codes[].encrypted_code` | `name`, `note` |
| `env-file` | `content` (dotenvx blob), `encrypted_dotenvx_key` | `label`, `tag`, `variable_count`, `project_id` |
| `audit-log` | — | who/action/resource/ip/ua/when only — **never** any secret |

¹ `password_hash` is `argon2id(authHash)` — not encrypted, but a one-way hash of an already-derived
value, so it is neither the password nor a replayable login token.

Server-side redaction ([`PLAN.md` §7.1](../PLAN.md)) keeps `password`, `token`, `authHash`,
`wrappedDEK`, `refreshToken` out of logs.

## 10. Divergences from the plan

The code is authoritative; note these differences from [`PLAN.md`](../PLAN.md):

- **`.env` blobs live in MongoDB, not Cloudinary.** The plan (§Locked Decisions, §6) specifies
  Cloudinary uploads with only a `path` reference in Mongo. The shipped code stores the encrypted
  `.env` document directly in a `content: String` field on the `env-file` model
  ([`env-file.model.ts:28`](../api/src/models/env-file.model.ts#L28)) and transports it base64 via the
  JSON body ([`env-file.service.ts:28`](../api/src/services/env-file.service.ts#L28)). There is no
  Cloudinary SDK call. The route comment "Cloudinary-backed encrypted blobs"
  ([`vault.routes.ts:47`](../api/src/routes/vault.routes.ts#L47)) is stale.
- **No multipart upload.** The `POST /env-files` endpoint is plain JSON (`content_b64`), not the
  multipart flow the plan sketches (§7).
- **Env crypto is native Rust throughout** (dotenvx crate), matching the plan's "no Node/dotenvx
  sidecar" intent — confirmed, not a divergence, but worth stating since the storage layer changed.
